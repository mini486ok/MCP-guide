FastMCP Tool 구현 & 스키마 매핑

이 가이드의 심장입니다. Python 함수의 어떤 부분이 MCP 서버의 어떤 필드로 변환되는가—이 매핑 규칙만 이해하면 좋은 Tool을 설계할 수 있습니다.

예상 학습 시간 · 30분 이 가이드의 핵심

AI가 실제로 보는 것

AI(Claude 등)는 내 Python 코드를 직접 보지 않습니다. MCP 프로토콜이 정해 놓은 JSON 형식의 정보만을 받습니다. 그 JSON은 대개 이런 형태입니다.

json
{
  "name": "hello",
  "description": "이름을 받아 인사말을 반환합니다.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "name": { "type": "string" }
    },
    "required": ["name"]
  },
  "outputSchema": {
    "type": "object",
    "properties": {
      "result": { "type": "string" }
    },
    "x-fastmcp-wrap-result": true
  }
}

이 JSON을 직접 쓰지는 않습니다. 우리가 쓰는 것은 그냥 Python 함수이고, FastMCP가 함수를 보고 위 JSON을 자동 생성합니다. 단, "어떤 Python 부분이 어떤 JSON 필드로 가느냐"를 알고 있어야 좋은 Tool을 짤 수 있습니다.

매핑 규칙 한눈에 보기

Python 쪽 MCP(JSON) 쪽 메모
함수 이름 def search_station(...) name 기본값. @mcp.tool(name="...")로 바꿀 수 있음.
Docstring 전체 description @mcp.tool(description="...")로 덮어쓸 수 있음.
파라미터 타입 힌트 (name: str) inputSchema.properties.name.type Pydantic이 JSON Schema를 생성.
기본값 없는 파라미터 inputSchema.required = None 또는 = 0 등 기본값이 있으면 required에서 제외.
Docstring의 Args: 블록 각 파라미터의 description Google 스타일 docstring 권장.
반환 타입 힌트 (-> int) outputSchema int/str/float 같은 원시 타입은 {"result": ...}로 감싸서 반환.
Annotated[str, Field(description=...)] 파라미터 설명을 더 정밀히 지정 docstring의 Args보다 우선 적용.
Pydantic BaseModel 반환 object 형식의 outputSchema 복잡한 응답일수록 이 방식 권장.
원칙

타입 힌트가 형식을 결정하고, docstring이 의미를 결정합니다. 두 가지를 성실히 적으면 그 자체로 AI가 이해하기 쉬운 Tool이 됩니다.

시각화: 같은 색 = 같은 필드

Python 함수
@mcp.tool
def find_train(
    departure: str,
    arrival: str,
    max_transfers: int = 1,
) -> list[dict]:
    """두 역 사이의 열차를 찾습니다.

    Args:
        departure: 출발역 이름.
        arrival: 도착역 이름.
        max_transfers: 허용하는 최대 환승 수.
    """
    ...
생성되는 JSON Schema
{
  "name": "find_train",
  "description": "두 역 사이의 열차를
     찾습니다...",
  "inputSchema": {
    "type": "object",
    "properties": {
      "departure": { "type": "string",
        "description": "출발역 이름." },
      "arrival":   { "type": "string",
        "description": "도착역 이름." },
      "max_transfers": { "type": "integer",
        "default": 1 }
    },
    "required": ["departure","arrival"]
  },
  "outputSchema": { "array of object" }
}

같은 색으로 칠해진 부분이 서로 짝입니다. 눈으로 한 번만 따라가 보세요.

패턴 1 — 가장 단순한 Tool

python
from fastmcp import FastMCP

mcp = FastMCP("example")

@mcp.tool
def km_to_miles(km: float) -> float:
    """킬로미터를 마일로 변환합니다.

    Args:
        km: 변환할 거리(킬로미터 단위).

    Returns:
        같은 거리의 마일 값.
    """
    return km * 0.621371
  • 이름: 함수명 km_to_miles가 그대로 Tool 이름이 됩니다.
  • 설명: docstring 첫 줄이 짧은 설명으로, 전체가 긴 설명으로 쓰입니다.
  • 입력 스키마: km이 number 타입의 필수 필드.
  • 출력 스키마: 원시 타입 float이라 {"result": 숫자} 형태로 감싸집니다.

패턴 2 — Annotated + Field 로 파라미터 설명 강화

docstring의 Args:로도 충분하지만, 제약 조건(최솟값·최댓값 등)을 함께 적고 싶으면 Annotated와 Pydantic의 Field를 씁니다.

python
from typing import Annotated
from pydantic import Field
from fastmcp import FastMCP

mcp = FastMCP("example")

@mcp.tool
def schedule_maintenance(
    station: Annotated[str, Field(description="점검할 역 이름")],
    hours: Annotated[int, Field(ge=1, le=24, description="점검 시간(1~24시간)")],
) -> str:
    """지정한 역의 점검 일정을 등록합니다."""
    return f"{station} 점검 {hours}시간 등록"

ge=1, le=24처럼 쓴 제약은 그대로 JSON Schema의 minimum / maximum으로 변환되어, AI가 "유효하지 않은 값은 아예 보내지 않게" 만듭니다.

패턴 3 — Pydantic BaseModel 반환

구조화된 여러 필드를 돌려주려면 Pydantic의 BaseModel이 가장 깔끔합니다. outputSchema가 정확하게 object 형태로 잡혀서 AI가 응답을 파싱하기 쉬워집니다.

python
from pydantic import BaseModel, Field
from fastmcp import FastMCP

mcp = FastMCP("example")

class Station(BaseModel):
    name: str = Field(description="역 이름")
    line: int = Field(description="호선 번호")
    lat: float
    lng: float

@mcp.tool
def get_station(name: str) -> Station:
    """역 이름으로 역 정보를 조회합니다.

    Args:
        name: 조회할 역의 이름(예: "서울역").
    """
    return Station(name=name, line=1, lat=37.5547, lng=126.9707)

이렇게 하면 반환값이 알아서 JSON으로 직렬화되고, outputSchema는 Station의 구조를 정확히 반영합니다. 여러 건을 돌려주고 싶다면 -> list[Station]으로 반환 타입만 바꾸면 됩니다.

파라미터 설계 상식

  • 기본값이 있으면 그 파라미터는 required에서 빠지고, 기본값도 스키마에 노출됩니다.
  • Optional(값 또는 None)이 필요하면 name: str | None = None 형태를 씁니다.
  • 열거형(정해진 선택지)은 Literal["A", "B"]를 쓰면 스키마에 enum으로 잡힙니다.
  • 너무 많은 파라미터(5개 이상)는 BaseModel 한 개로 묶어 하나의 파라미터로 받는 게 깔끔합니다.

반환 타입 실전 선택 가이드

무엇을 돌려주나추천 반환 타입결과
숫자 하나-> int / -> float{"result": 42}로 감싸짐
문자열 하나-> str{"result": "..."}로 감싸짐
성공/실패-> bool{"result": true}
구조화된 하나의 객체-> Station (BaseModel)object로 정확히 매핑
목록-> list[Station]array of object
간단한 키-값-> dict스키마는 느슨해짐. 가능하면 BaseModel 권장
이런 것은 피하세요

제너레이터(yield)나 스트림 객체를 직접 반환하면 일반 Tool에서는 처리되지 않습니다. 또한 Any 반환은 AI가 스키마를 신뢰하지 못해 품질이 떨어집니다. 반환 타입은 항상 명시하세요.

예외와 오류 처리

에러가 났을 때는 값을 억지로 만들어 돌려주기보다 그냥 예외를 던지세요. FastMCP가 알아서 MCP 프로토콜의 에러 응답으로 변환해 줍니다.

python
@mcp.tool
def get_station(name: str) -> Station:
    """역 이름으로 역 정보를 조회합니다."""
    if name not in STATIONS:
        raise ValueError(f"'{name}' 역을 찾을 수 없습니다.")
    return STATIONS[name]

예외 메시지는 그대로 AI가 보기 때문에, 사람이 읽을 수 있는 한국어(또는 영어) 문장으로 써 주세요. "VALIDATION_ERR_NPE_42" 같은 코드는 AI에게 도움이 되지 않습니다.

실습: 여러 언어로 인사하는 Tool

실습 — 10분

앞 장에서 만든 hello Tool을 "여러 언어로 인사하는" 버전으로 확장해 봅니다. Literal로 선택지를 제한하고, BaseModel로 결과를 구조화합니다.

python · hello_multi.py
from typing import Literal
from pydantic import BaseModel
from fastmcp import FastMCP

mcp = FastMCP("hello-multi")


class Greeting(BaseModel):
    language: str
    text: str


GREETINGS: dict[str, str] = {
    "ko": "안녕하세요, {name}님!",
    "en": "Hello, {name}!",
    "ja": "こんにちは、{name}さん!",
}


@mcp.tool
def greet(
    name: str,
    language: Literal["ko", "en", "ja"] = "ko",
) -> Greeting:
    """지정한 언어로 인사말을 만들어 돌려줍니다.

    Args:
        name: 인사할 대상의 이름.
        language: 인사말의 언어 코드. ko, en, ja 중 하나.
    """
    template = GREETINGS[language]
    return Greeting(language=language, text=template.format(name=name))


if __name__ == "__main__":
    mcp.run()

실행한 뒤 다음 장에서 배울 MCP Inspector로 확인해 보면, language 필드가 enum: ["ko", "en", "ja"]로 잡혀 있고 Greeting 객체 형식의 outputSchema가 자동 생성된 것을 볼 수 있습니다.

이 장의 핵심 한 줄

"타입 힌트 + docstring + (필요하면) BaseModel" — 이 세 가지만 성실히 적으면, 스키마는 FastMCP가 알아서 아주 정확하게 만들어 줍니다.
