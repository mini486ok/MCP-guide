스키마 확인·수정

내가 만든 MCP 서버가 AI에게 실제로 어떻게 보이는지, AI에게 전달되는 description·inputSchema·outputSchema를 눈으로 직접 확인하고 고치는 방법을 배웁니다.

예상 학습 시간 · 15분 Node.js 필요

왜 검증이 필요할까?

코드만 봐서는 "AI가 이 Tool을 어떻게 이해할지" 감을 잡기 어렵습니다. 실제로 눈으로 보면 많은 것이 달라집니다.

  • docstring을 빼먹어서 description이 비어 있을 수 있습니다.
  • 타입을 int로 썼는데 의도는 float이었을 수 있습니다.
  • 반환 타입을 지정하지 않아 outputSchema가 생성되지 않을 수 있습니다.

이런 문제를 "AI가 이상하게 동작해서 알게" 되는 대신, 서버를 연결하기 전에 우리가 직접 볼 수 있게 해 주는 도구가 MCP Inspector입니다.

MCP Inspector란?

Anthropic의 MCP 팀이 공식 제공하는 개발자용 웹 UI입니다. 브라우저에서 우리 서버에 연결해:

  • 노출된 Tool/Resource/Prompt 목록을 확인하고
  • 각 Tool의 설명·입력 스키마·출력 스키마를 JSON으로 살펴보고
  • 실제로 Tool을 호출해 응답을 바로 확인

할 수 있습니다. 한 마디로 "MCP 서버용 Postman"이라고 생각하면 됩니다.

설치 · 실행

MCP Inspector는 Node.js 기반입니다. 먼저 Node.js가 필요합니다.

Step 1. Node.js 설치

  1. nodejs.org에서 LTS 버전 다운로드 → 설치(기본 설정 그대로).
  2. 설치 확인:
bash
node --version
npx --version

Step 2. Inspector로 서버 열기

두 가지 방법이 있습니다. 초보자에게는 방법 A가 훨씬 권장입니다.

방법 A — FastMCP의 dev 명령 (권장)

FastMCP 자체가 Inspector를 함께 띄워 주는 편리한 커맨드를 제공합니다.

bash
fastmcp dev hello.py

이 한 줄로:

  1. Inspector를 자동으로 다운로드·실행합니다.
  2. 내 서버(hello.py)를 그 Inspector에 연결합니다.
  3. 브라우저가 자동으로 열립니다(열리지 않으면 터미널에 출력된 URL을 직접 여세요. 보통 http://localhost:6274).

방법 B — Inspector를 직접 실행

bash
npx @modelcontextprotocol/inspector python hello.py

이때도 브라우저가 뜨고, 왼쪽에서 "Connect" 버튼을 누르면 우리 서버와 연결됩니다.

화면 사용법

Inspector는 크게 좌측 네비게이션과 우측 상세 뷰로 나뉩니다. 한 번만 익혀두면 매일 쓰게 됩니다.

좌측 Tools 탭에서 Tool을 고르면, 오른쪽에 스키마와 테스트 영역이 나타납니다.

Step 1. Tools 탭 열기

좌측 네비게이션에서 "Tools"를 클릭한 뒤, 중간 목록에서 보고 싶은 Tool을 선택합니다. 04장에서 만든 hello를 고른다고 가정해 봅시다.

Step 2. 스키마 확인

우측에 다음 세 가지가 JSON으로 보입니다.

  • Description — 여러분의 docstring이 그대로 들어가 있는지 확인.
  • Input Schema — 파라미터 이름·타입·필수 여부·설명.
  • Output Schema — 반환 타입의 구조(없으면 이 섹션이 통째로 빠져 있음).

Step 3. 호출해 보기

같은 탭 아래쪽에 Tool을 직접 실행해 볼 수 있는 폼이 있습니다. name에 "미나"를 넣고 "Run"을 누르면, 실제 응답이 JSON으로 표시됩니다.

스키마가 기대와 다를 때: 원인과 처방

대부분의 문제는 아래 표 한 장으로 해결됩니다.

증상원인고치는 법
description이 비어 있음 함수에 docstring이 없음 함수 바로 아래에 """...""" 추가
파라미터 description이 비어 있음 docstring에 Args: 블록이 없음 Google 스타일의 Args: 추가 또는 Annotated[..., Field(description=...)]
파라미터 type이 string인데 숫자를 기대 타입 힌트가 str 또는 미지정 age: int처럼 정확히 지정
모든 파라미터가 required로 잡힘 기본값이 없어서 선택 인자에 = None 또는 구체적 기본값 부여
outputSchema가 없음 반환 타입 힌트(-> ...)가 없음 반환 타입을 명시. 복잡하면 BaseModel 도입
반환 스키마가 "그냥 object"처럼 느슨함 dict나 Any를 반환 Pydantic BaseModel 서브클래스로 교체
호출 시 Validation error 입력이 스키마를 어김 Inspector 폼의 에러 메시지를 읽어 값 수정(이게 Inspector의 진짜 가치)

바뀌면 Inspector를 다시 열어야 하나?

fastmcp dev로 실행했다면, 코드를 저장하자마자 자동으로 서버가 재시작됩니다. Inspector 화면에서 해당 Tool을 다시 클릭하기만 하면 최신 스키마가 보입니다. 자동 재시작이 안 된다면 터미널을 Ctrl + C로 끝낸 뒤 다시 실행하세요.

CI/회귀 검증: CLI 모드

Inspector는 브라우저 UI뿐 아니라 명령행(CLI)에서도 동작합니다. 팀에서 Tool 스키마가 바뀌지 않는지 지속적으로 확인하고 싶을 때 유용합니다.

bash
# 모든 Tool의 목록과 스키마를 JSON으로 출력
npx @modelcontextprotocol/inspector --cli python hello.py --method tools/list

# 특정 Tool을 주어진 인자로 호출해 결과를 출력
npx @modelcontextprotocol/inspector --cli python hello.py \
    --method tools/call --tool-name hello --tool-arg name=미나

이 출력을 jq로 확인하거나 GitHub Actions에서 스냅샷 비교 테스트에 활용할 수 있습니다. 개인 개발 단계에서는 여기까지 하지 않아도 됩니다.

개발 루프 제안

실무 개발은 다음 루프를 반복합니다. ① 함수 수정 → ② 저장 → ③ Inspector에서 새로고침 → ④ 스키마/응답 확인 → ⑤ 수정. 이 루프가 몸에 배면 MCP 서버를 만드는 속도가 크게 빨라집니다.
